# B4RRHH - Initial Architecture Decisions

## Repositories
- The project uses separate repositories:
  - b4rrhh-backend
  - b4rrhh-frontend

## General architecture
- Strict hexagonal architecture
- Contract-first approach with OpenAPI
- Backend and frontend are autonomous
- OpenAPI source of truth lives in backend
- Frontend uses a derived copy and must not edit it manually

## Backend organization
- Package structure is organized by business capability first, then by layer
- Initial backend modules:
  - employee
  - rulesystem
  - shared

## Technology choices
- Java 21
- Spring Boot (latest stable used by project)
- Maven
- PostgreSQL
- Flyway
- JPA/Hibernate in infrastructure only
- Spring Security with simple role model

## Initial security roles
- ADMIN
- HR_MANAGER
- HR_VIEWER

## Functional scope
Current scope is Personnel Administration for Spain only.

Included in pilot:
- employees
- personal data
- company presence (hire/termination periods)
- employment contracts
- rule-system driven master entities

Excluded:
- payroll
- legal reporting
- speculative future modules

## Domain boundaries
- employee module contains employee-related business facts
- rulesystem module contains parameterized entities and catalog-like structures
- rulesystem does not model employee facts
- employee facts may reference rule-system entities

## Core employee domain concepts
- Employee
- EmployeePersonalData
- EmployeeCompanyPresence
- EmploymentContract

## Core rule system concepts
- RuleSystem
- RuleEntityType
- RuleEntity
- RuleEntityRelation

## Modeling principles
- employee has a technical numeric autogenerated primary key
- employee has a business employeeNumber as a string
- employeeNumber is unique and distinct from database primary key
- historical information should be modeled with validFrom / validTo
- business history should not be deleted as a normal operation

## Presence and contract relationship
- EmployeeCompanyPresence is a core historical business fact
- EmploymentContract is another employee business fact
- EmploymentContract does not need to depend structurally on EmployeeCompanyPresence in v0
- Business rules must ensure that contracts and similar facts are temporally consistent with an existing valid company presence
- presenceId may exist as a useful reference where needed, but it is not the mandatory structural parent of all dependent facts

## Copilot usage policy
- Copilot may generate scaffolding, refactorings, and tests
- Copilot must not invent labor-law rules or HR business rules
- Copilot must not alter the data model without explicit approval
- Copilot must not mix layers
