# Spec: Nómina — Gestión de Recibos (Viewer + State Transitions)

**Date:** 2026-04-24  
**Status:** Approved by user  
**Scope:** Frontend `b4rrhh_frontend` + Backend gaps in `b4rrhh_backend`

---

## 1. Scope

A new "Recibos" entry under the existing "Nómina" menu that allows HR to:

- **A · View** calculated payrolls filterable by period, employee number/name, and status.
- **B · Inspect** a payroll as a traditional Spanish *recibo de nómina* (folio layout).
- **C · Transition state**: Invalidar and Validar from CALCULATED; Recalcular from any recalculable state.

**Out of scope:** Calculation run management (launching batch runs). That is handled elsewhere.

## 3. Layout

### 3.1 Left panel (300 px, fixed)

- **Filters** (free combination, all optional):
  - Período — text input `yyyyMM`
  - Estado — dropdown with values `Todos | CALCULATED | NOT_VALID | EXPLICIT_VALIDATED | DEFINITIVE`
  - Empleado — free text (employee number or name prefix)
  - Buscar button
- **Results list**: one row per payroll, showing:
  - Employee number (bold when selected)
  - Status badge (colour-coded — see §5)
  - Period + payroll type line
  - Blue left-border indicator on selected row
- **Footer**: `N nóminas encontradas`

### 3.2 Right panel (flex-1, grey background `#d1d5db`)

- **Action bar** (dark background `#1e1e2e`): employee key, status badge, state-transition buttons (see §5).
- **Folio** (white, max-width 780 px, left-aligned, paper shadow): the payslip document (see §4).

### 4.2 Concept table — 7 columns

| Column | Backend field | Null treatment |
|---|---|---|
| Período | `originPeriodCode` | — if null |
| Clave | `conceptCode` | — |
| Concepto | `conceptLabel` | — |

| Deducciones | `amount` (if nature = DEDUCTION or future TOTAL_DEDUCTIONS) | — |

Rows are ordered by `displayOrder` ascending. The frontend performs no arithmetic.

**Rendering rules by `conceptNatureCode`:**

| conceptNatureCode | Devengos col | Deducciones col | Row style |
|---|---|---|---|
| `EARNING` | amount | — | Normal |
| `DEDUCTION` | — | amount | Normal |
| `BASE` | — | — | Normal (informational) |
| `INFORMATIONAL` | — | — | Normal (informational) |
| *(future)* `TOTAL_EARNINGS` | amount | — | **Bold, thick top border** |
| *(future)* `TOTAL_DEDUCTIONS` | — | amount | **Bold** |
| *(future)* `NET_PAY` | colspan — rendered as footer | — | Dark footer (§4.3) |

> **Note:** `TOTAL_EARNINGS`, `TOTAL_DEDUCTIONS`, and `NET_PAY` are planned extensions to the `FunctionalNature` enum in the payroll engine. Exact enum value names are TBD and will be defined when aggregate concept generation is implemented. The frontend rendering logic must be keyed to agreed string constants that match `FunctionalNature.name()`.

### 4.3 Footer — Líquido a Percibir

Dark bar (`#212529`), white text. The net-pay amount is rendered from the backend `NET_PAY` concept line. Until that concept is generated by the engine, the footer shows a placeholder or is omitted.

## 5. State Transitions

### 5.1 Valid transitions

| From state | Action | To state | Button label | Button colour |
|---|---|---|---|---|
| `CALCULATED` | Invalidar | `NOT_VALID` | Invalidar | Red |
| `CALCULATED` | Validar | `EXPLICIT_VALIDATED` | Validar | Blue |
| `NOT_VALID` | Recalcular | `CALCULATED` (engine decides) | Recalcular | Amber |

**Rule: `NOT_VALID` cannot transition directly to any valid state.** Only the calculation engine can promote a `NOT_VALID` payroll to `CALCULATED` via Recalcular. The backend endpoint enforces this — it rejects recalculate requests for any state other than `NOT_VALID`.

### 5.2 Button visibility per state

| State | Invalidar | Validar | Recalcular |
|---|---|---|---|
| `NOT_VALID` | — | — | ✓ |
| `CALCULATED` | ✓ | ✓ | — |
| `EXPLICIT_VALIDATED` | — | — | — |
| `DEFINITIVE` | — | — | — |

### 5.3 Status badge colours

| Status | Background | Label |
|---|---|---|
| `CALCULATED` | Green `#a6e3a1` | CALCULADA |
| `NOT_VALID` | Red `#f38ba8` | INVÁLIDA |
| `EXPLICIT_VALIDATED` | Blue `#89b4fa` | VALIDADA |
| `DEFINITIVE` | Purple `#cba6f7` | DEFINITIVA |

## 6. Backend Gaps

These backend changes are required before the frontend can be fully functional:

### 6.1 Payroll list/search endpoint (Priority: HIGH — blocks feature A)

No search endpoint exists. The current API only exposes a single-record fetch by full business key. A new endpoint is needed:

```
GET /payrolls?periodCode=&employeeNumber=&status=
```

Returns a paginated or flat list of payroll summaries (no concept lines — only header fields and status).

### 6.2 State transition endpoints (Priority: HIGH — blocks feature C)

New endpoints needed:
- `POST /payrolls/{ruleSystemCode}/{employeeTypeCode}/{employeeNumber}/{payrollPeriodCode}/{payrollTypeCode}/{presenceNumber}/invalidate` → `NOT_VALID`
- `POST /payrolls/{...}/validate` → `EXPLICIT_VALIDATED`
- `POST /payrolls/{...}/recalculate` → triggers engine, result is `CALCULATED`

### 6.3 Aggregate concept generation (Priority: MEDIUM — blocks TOTAL and Líquido rows)

The payroll engine does not currently generate `TOTAL_EARNINGS`, `TOTAL_DEDUCTIONS`, or `NET_PAY` concept lines. Until it does:
- The TOTAL row at the bottom of the Devengos/Deducciones columns will be absent.
- The Líquido footer will be absent or show a placeholder.

Steps required:
1. Extend `FunctionalNature` enum with the agreed aggregate values (names TBD).
2. Engine generates the closing concept lines as part of each payroll calculation run.

## 7. Frontend Architecture

Follows the existing Angular 21 feature pattern:

```
src/app/features/nomina/recibos/
  recibos.routes.ts           — lazy route definition
  recibos-page/
    recibos-page.component.ts — master/detail shell
    recibos-page.store.ts     — signals store: filters, list, selected payroll, concept lines
  recibos-list/
    recibos-list.component.ts — left panel: filters + scrollable list
  recibos-detail/
    recibos-detail.component.ts     — right panel: action bar + folio
    recibos-folio.component.ts      — the payslip document itself
  gateway/
    recibos.gateway.ts        — API calls (list, fetch concepts, transitions, recalculate)
  model/
    payroll-summary.model.ts  — domain model for list items
    payroll-detail.model.ts   — domain model for concept lines
```

The store holds:
- `filters` signal (period, employee, status)
- `payrolls` signal (list of summaries)
- `selectedPayroll` signal
- `concepts` signal (concept lines for selected payroll)
- `loading` / `error` derived signals

No arithmetic is performed in any component, store, or gateway.
